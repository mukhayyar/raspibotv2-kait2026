Adaptive Hierarchical Detection: A Two-Phase Framework for Context-Aware Object Detection in Edge Devices

Muhammad Tsaqif Mukhayyar  ·  Kosuke Takano  ·  Idris Winarno
Kanagawa Institute of Technology (KAIT), Atsugi, Japan  |  Politeknik Elektronika Negeri Surabaya (PENS), Indonesia
PENS-KAIT Joint Research Programme · March 2026
Abstract Real-time object detection on resource-constrained edge devices demands careful balancing of computational budget, semantic precision, and adaptability to dynamic environments. This paper presents Adaptive Hierarchical Detection (AHD), a two-phase framework that couples lightweight scene recognition with open-vocabulary YOLO-World detection to achieve context-aware inference on a Raspberry Pi 5. Phase 1 uses a GoogLeNet CNN pre-trained on Places365 (365 scene categories) to identify the current environment at 1 Hz. Phase 2 dynamically scopes the YOLO detection vocabulary to a scene-specific class subset retrieved from a SQLite context database, reducing the open-vocabulary CLIP embedding space and improving detection relevance. We evaluate two class vocabularies—COCO-80 and Objects365—across 365 scenes and benchmark inference latency across four model serialization formats (PyTorch, ONNX, TFLite float32, TFLite float16) and a Vulkan GPU path. TFLite float16 achieves the lowest Phase 2 latency at 302 ms. Context switching in Classes Mode incurs 3–8 s due to CLIP text encoding; Model Mode reduces this to 300–600 ms at the cost of 50 MB disk per scene. The system is validated in a live robot teleoperation platform (Yahboom RaspBot V2) with full web dashboard, sensor integration, and MJPEG/WebRTC streaming.
Keywords: edge inference · context-aware detection · YOLO-World · Places365 · Raspberry Pi 5 · two-phase detection · adaptive vocabulary · embedded AI · robot teleoperation

1. Introduction

Deploying real-time object detection on edge devices—single-board computers, embedded robotics platforms, and IoT nodes—presents a fundamental tension: general-purpose detectors must recognize hundreds of object classes, yet the device's computational budget is severely constrained. A fixed 80-class COCO detector running on a Raspberry Pi 5 spends equal inference capacity on detecting surfboards in a hospital corridor as it does on detecting wheelchairs and medical equipment—a semantic misallocation that both degrades accuracy and wastes computation.

This paper addresses this tension through Adaptive Hierarchical Detection (AHD), a two-phase framework that leverages scene context to dynamically focus the object detection vocabulary on semantically relevant classes. The framework is implemented in the PENS-KAIT 2026 system—a full-stack robot teleoperation platform running on a Yahboom RaspBot V2 (Raspberry Pi 5) with a React-based web dashboard, Socket.IO control interface, MJPEG/WebRTC streaming, and hardware control over motors, servos, LEDs, and sensors.

The core insight is that scene context provides strong prior information about which object classes are likely to be present. A kitchen is likely to contain cups, bottles, and appliances; an office is likely to contain laptops, monitors, and chairs. By conditioning Phase 2 detection on Phase 1 scene recognition, AHD reduces both the vocabulary search space and the cognitive load of the detection output, improving relevance without retraining the detector.

Contributions of this paper:

  1. A complete two-phase detection framework integrating Places365 scene recognition with dynamic YOLOv8-World class scoping, deployed on Raspberry Pi 5.
  2. A dual-vocabulary context database mapping 365 Places365 scenes to both COCO-80 and Objects365 class subsets, with empirical comparison of detection breadth.
  3. A systematic benchmark of four inference formats (PyTorch, ONNX, TFLite float32, TFLite float16) and NCNN Vulkan on VideoCore VII GPU, with root-cause analysis.
  4. A production-grade deployment integrating detection, streaming, hardware control, and authentication in a containerized web application.

2. System Architecture Overview

The PENS-KAIT 2026 system implements a layered, multi-threaded architecture. 3. Phase 1: Scene Recognition

Phase 1 provides the environmental context that drives vocabulary selection in Phase 2. It runs as a dedicated daemon thread throttled to 1 Hz to avoid CPU contention with the YOLO inference thread.

3.1 Model Architecture

We use a GoogLeNet (Inception v1) CNN pre-trained on Places365 [1], loaded via OpenCV's DNN module (cv2.dnn.readNetFromCaffe). The model outputs a 365-dimensional softmax probability vector over scene categories. Input frames are resized to 224×224 and mean-subtracted with (104, 117, 123) in BGR space.

3.2 Scene Stability Tracking (Debounce)

Raw scene predictions are noisy due to motion blur, lighting changes, and frame-to-frame variance. AHD implements a scene stability tracker that requires the same scene to be predicted for N = 3 consecutive frames before triggering a context switch. This debounce mechanism prevents thrashing while keeping adaptation latency under 3 seconds at 1 Hz polling.

Figure 2. Phase 1 inference pipeline. The GoogLeNet scene recognizer runs at 1 Hz; a stability counter debounces predictions before triggering a Phase 2 context switch.

4. Phase 2: Adaptive Object Detection

4.1 YOLOv8-World Architecture

Phase 2 uses YOLOv8s-WorldV2, an open-vocabulary extension of YOLOv8 that replaces the fixed classification head with a CLIP-conditioned embedding head. At inference time, text prompts of the form "a photo of a {class_name}" are encoded by a CLIP text transformer into 512-dimensional embedding vectors, which are injected into the detection head to produce class probabilities.

P(c | b) = softmaxc ∈ Cs ( vb · ec / τ )

where vb is the visual region embedding of bounding box b, ec is the CLIP text embedding of class c, and τ is the temperature parameter. The class set Cs is the scene-specific subset retrieved from the context database, reducing the denominator from 365 potential classes to 11–40 per scene.

4.2 Context Switching Modes

Classes Mode (Default)

Calls model.set_classes(C_s) which runs the CLIP text encoder on all class names in the new scene's vocabulary. This encoding takes 3–8 seconds on the Pi 5 CPU due to the CLIP transformer forward pass. No disk storage required beyond the base model.

Model Mode (Fast)

Loads a pre-generated frozen .pt checkpoint with baked-in class embeddings for a specific scene. Load time is 300–600 ms (10× faster). Trade-off: ~50 MB per frozen model × 365 scenes ≈ 18 GB total disk storage required.

5. Context Management Architecture

The context manager bridges scene recognition (Phase 1) and object detection (Phase 2). It maintains a SQLite database mapping each of the 365 Places365 scenes to an appropriate subset of YOLO detection classes, under two vocabularies: COCO-80 (for compatibility with fixed-class models) and Objects365 (for broader semantic coverage).

5.2 Class Assignment Algorithm

Classes are assigned to scenes via two complementary lookup mechanisms:

  1. Category-level lookup: The Places365 CSV provides 16 binary Level-2 category flags per scene. Each category maps to a pre-defined class list. If flag i is set, all classes from CATEGORY_CLASSES[i] are added.
  2. Keyword-level lookup: The scene name (e.g., living_room_outdoor) is tokenized and each token is matched against a KEYWORD_CLASSES dictionary (100+ entries).
  3. Validation: All candidate classes are filtered against the target vocabulary (COCO-80 or Objects365), then deduplicated and sorted.

9. Discussion

9.1 Why TFLite float16 Outperforms ONNX on ARM

The XNNPACK delegate provides hand-written ARM NEON/FP16 convolution kernels tuned at the register level. For depthwise convolution—dominant in YOLO's CSP bottleneck—XNNPACK achieves near-theoretical NEON throughput. ONNX Runtime's CPUExecutionProvider uses the more general MLAS backend without these aarch64-specific paths. The marginal gain of float16 over float32 (~2%) is attributed to reduced L2/L3 cache pressure: float16 weights occupy half the memory bandwidth while XNNPACK still computes in float32.

9.2 Root Cause of NCNN Vulkan Failure

Four compounding factors make NCNN Vulkan non-viable on VideoCore VII:

  1. Missing fp16 accumulation: NCNN convolution kernels rely on 16-bit accumulation; its absence forces a SPIR-V fallback that doubles register pressure.
  2. Single compute queue: Prevents overlapping memory transfers with computation, serializing every vkQueueSubmit.
  3. No cooperative matrix extension (VK_KHR_cooperative_matrix): Forces scalar/vector GEMM paths for fully-connected layers and 1×1 convolutions.
  4. SPIR-V JIT compilation (Mesa v3dv): Cold-start compilation of ~150 unique shader variants contributes >10 s latency; cache invalidation explains the second-run regression (15 s → 47 s).

9.3 Classes Mode vs. Model Mode Trade-off

The 3–8 second CLIP encoding overhead of Classes Mode is prohibitive for reactive robotics. Model Mode resolves this at the cost of disk space (~18 GB for all 365 scenes as frozen models). A practical middle ground is a hot-cache strategy: pre-generate frozen models for the 20–30 most common deployment scenes, falling back to Classes Mode for rare scenes. This reduces disk usage to ~1.5 GB while keeping switches under 600 ms for >90% of real-world environments.

9.4 Objects365 vs. COCO-80

Objects365 provides on average 1.97× more semantically relevant classes per scene. For indoor scenes with specialized equipment (hospital: +167%, concert: ∞), the gain is dramatic. For outdoor transport scenes (street: +58%), the gain is moderate but meaningful. The trade-off is potential detection noise: more candidate classes increase the chance of false positives at low confidence thresholds. We recommend a confidence threshold of 0.35–0.45 for Objects365 mode vs. 0.25–0.35 for COCO-80.

9.5 Practical Deployment Recommendations

  1. Use TFLite float16 for YOLO26n (fixed-class scenes): 302 ms, 3.2 FPS, 34% faster than PyTorch.
  2. Use YOLOv8s-WorldV2 in Model Mode for open-vocabulary scenes: 450 ms switch + 240 ms inference.
  3. Do not use NCNN Vulkan on Pi 5 VideoCore VII for neural inference.
  4. Use N=3 stability threshold for Phase 1 scene debounce; increase to N=5 in high-vibration environments.
  5. For sub-100 ms inference, add Hailo-8L AI HAT+ (26 TOPS via PCIe M.2).
  6. Use WebRTC where latency is critical; MJPEG for broader compatibility.

10. Conclusion & Future Work

We presented Adaptive Hierarchical Detection (AHD), a two-phase framework that combines GoogLeNet scene recognition (Places365, 1 Hz) with dynamic YOLOv8-World class scoping (3–5 Hz) to achieve context-aware object detection on Raspberry Pi 5. The framework achieves 100% semantic relevance of detected classes by restricting the open-vocabulary search space to scene-appropriate subsets drawn from a dual-vocabulary SQLite context database (COCO-80 and Objects365, 365 scenes each).

Our inference benchmark establishes TFLite float16 + XNNPACK as the optimal CPU format at 302 ms (3.2 FPS), outperforming PyTorch by 34% and ONNX by 28%. NCNN Vulkan on VideoCore VII is non-viable at 15,000–47,000 ms due to structural GPU limitations. Context switching in Classes Mode (3–8 s) can be reduced to 300–600 ms via Model Mode at a disk cost of ~50 MB/scene.

The complete system is deployed and validated on a Yahboom RaspBot V2, integrating motor control, servo, LED, buzzer, sensor monitoring, and MJPEG/WebRTC streaming in a production Docker container.

Future Work

  • Hailo-8L NPU integration: Target <10 ms inference via PCIe AI HAT+ for real-time Phase 2 at 30 FPS.
  • Quantized Phase 1: Export GoogLeNet to TFLite INT8 to reduce Phase 1 CPU cost from ~50 ms to ~10 ms.
  • Hot-cache frozen models: Pre-generate top-30 scene models to bring Model Mode disk usage to ~1.5 GB.
  • Adaptive threshold tuning: Auto-calibrate confidence thresholds per scene based on historical false-positive rates.
  • SLAM integration: Combine AHD with ORB-SLAM3 for persistent scene maps that pre-warm context before entering a new room.
  • Multi-label scene recognition: Replace top-1 scene with top-K union for transitional environments (e.g., kitchen-living_room open plans).
