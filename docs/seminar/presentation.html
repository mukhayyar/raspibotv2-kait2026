3 · Problem Research question 4 / 14

Problem Statement

Existing mobile robot platforms treat teleoperation and computer vision as separate, statically configured systems.

❌ Problem 1 — Static Detection

Standard YOLO models use fixed class sets. Moving the robot from a kitchen to an office doesn't change what it detects — irrelevant classes waste computation and increase false positives.

❌ Problem 2 — COCO-80 Vocabulary Ceiling

COCO-80 covers general objects but misses scene-specific categories. Hospital scenes have no stethoscope, office scenes have no whiteboard — the vocabulary limits detection utility.

❌ Problem 3 — Edge Deployment Constraints

Real-time context switching on Raspberry Pi 5 CPU must compete with inference, sensor polling, and WebSocket broadcasting — all on a single ARM board with no GPU.

RQ 1

Can scene recognition automate YOLO class selection?
Can GoogLeNet Places365 reliably trigger correct COCO-80 class subsets per scene, achieving switches within an acceptable latency window?

RQ 2

What is the actual latency of set_classes() on edge hardware?
On Raspberry Pi 5 ARM Cortex-A76, how does context switch latency scale with the number of target COCO-80 classes?

RQ 3

Is the 3-second switch target achievable with COCO-80 classes mode?
Given the observed latency-vs-classes relationship, for which scene configurations does the system meet or exceed the target?

4 · Method Phase 2 — Context-aware YOLO switching 7 / 14

Phase 2: COCO-80 Context Switching

When Phase 1 declares a stable scene, _switch_scene() queries ContextManager and reconfigures YOLOWorld v2 with the scene's COCO-80 class subset.

YOLOWorld v2 — Classes Mode

yolov8s-worldv2.pt encodes class names as CLIP text embeddings. Calling set_classes(["person","chair","laptop",...]) re-computes embeddings for all target classes — this is the dominant cost on CPU.

Timing Breakdown (per switch)

DB lookup (db_query_ms) 0.3–4 ms
set_classes() CLIP encoding 620–12,132 ms
Total average 2,703 ms

⚠️ Finding: Latency Scales with Class Count

CLIP computes one embedding per class. More target classes = more inference calls = higher set_classes() cost. This was not anticipated from documentation.

5 · Results Phase 2 switch latency analysis 9 / 14

Phase 2 Switch Latency — Real Measurements

Source: access.db → inference_history · 3,355 total switches · classes mode · YOLOWorld v2 + COCO-80 · Raspberry Pi 5 CPU

Answer to RQ2 — Latency is Class-Count Dependent

CLIP text embedding computation in set_classes() scales with the number of target classes. On Raspberry Pi 5 ARM CPU (no GPU), this costs ~280 ms per class at typical workloads.

Answer to RQ3 — 3 s Target Partially Met

Scenes with ≤10 classes (65% of switches) meet the 3-second target on average. Scenes with ≥12 classes (conference, airplane, food court) consistently exceed it.

All 3,355 switches summary
Min  0.0 ms (cold cache)
Avg  2,703 ms
Max  12,132 ms
Over 3 s  33.5%

DB Lookup Is Negligible

db_query_ms is consistently 0.3–4 ms — the SQLite lookup with B-tree index is not the bottleneck. All latency comes from CLIP embedding computation in set_classes().

Primary Open Challenge

Phase 2 switch latency for scenes with >10 COCO-80 classes exceeds the 3-second target on Raspberry Pi 5 CPU. The bottleneck is set_classes() CLIP embedding computation, not database access or YOLO inference.

Teleoperation: All Targets Met

Control latency, sensor polling, camera FPS, and Phase 1 inference all meet or exceed targets. The robot is fully controllable during Phase 2 switch operations — switching runs in a separate thread and does not block movement commands.

DB Lookup: Negligible

db_query_ms consistently 0.3–4 ms across all 3,355 switches. The ContextManager's B-tree index on scene_name is not the bottleneck at any scale.

6 · Future Work Next steps 13 / 14

Future Work

⚡

Reduce set_classes() Latency

The primary bottleneck: CLIP embedding computation for COCO-80 classes on CPU. Strategies: pre-compute and cache all 80 COCO embeddings at startup, batch encode only changed classes, or offload to GPU.

Highest Priority
🔥

Vulkan GPU Acceleration

Raspberry Pi 5 VideoCore VII supports Vulkan 1.2 compute. Offloading CLIP encoding and YOLO inference to GPU could reduce Phase 2 switch latency from ~2.7 s to target <500 ms. Exploratory work in vulkan_test.py.

High Priority
💾

NVMe SSD Procurement

M.2 HAT+ with NVMe SSD enables full on-device deployment of all 365 scene-specific models and the complete benchmark suite — removing the Google Drive dependency for production use.

High Priority
🎯

Optimise COCO-80 Class Subsets

Reduce the number of classes per scene to the minimum semantically meaningful set. Scenes with ≤8 classes achieve <2 s switches — pruning food_court from 17 to 8 classes would cut its avg from 4.7 s to ~2 s.

Medium Priority
📡

WebRTC Production Hardening

Complete ICE/TURN configuration for non-LAN use. Formalise aiortc error recovery and connection state management to make WebRTC the default stream transport.

Medium Priority
📐

Full 365-Scene Benchmark

Complete latency profiling for all 365 Places365 scenes with full COCO-80 class mappings. Establish a per-scene latency budget and flag scenes requiring class-count reduction. In progress on Google Colab.

In Progress
